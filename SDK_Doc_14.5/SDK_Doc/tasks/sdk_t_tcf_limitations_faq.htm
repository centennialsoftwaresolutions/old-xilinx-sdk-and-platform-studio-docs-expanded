Known Issues and Limitations

In which scenario I should use Xilinx TCF debugger, Xilinx C/C++ Debugger(XMD+GDB).

It is preferable to use Xilinx TCF debugger for debugging ARM applications and use Xilinx C/C++ Debugger (XMD+GDB) to debug Microblaze applications, if you are connected with “Digilent” cable.

Debug Stack Trace is not showing automatically as with the Xilinx customized TCF Debugger.

Expand the tree and choose the correct processor core to see the debug stack trace.

Cleaning/importing old BSP projects results in build error(or) Modifying the BSP settings does not compile automatically.

Workaround for this issue is to build all applications once again using Run > Build All or Ctrl+B.

Note: BSP project that is created with the latest version of SDK will not have this problem.

Automatic perspective switch from C/C++ to Debug perspective is not happening with Xilinx customized TCF debugger.

Workaround for this issue is to either:

  • Go to Window > Preferences > Run/ Debug > Perspectives > Open the associated perspective when launching > Always
  • Manually switch from C/C++ to Debug perspective by clicking Window > Open Perspective > Debug.

Debug stack trace is not showing automatically as shown below with Xilinx customized TCF debugger.

Expand the tree and choose the correct processor Core to see the debug stack trace.

Watchpoints that are set from the Outline View are not working with Xilinx customized TCF debugger.

Set watch points either from editor (Right-click on a variable > Add Watchpoint) or from Breakpoints view.

Global variables are not displayed with Xilinx Customized TCF debugger.

Hover the mouse on the global variable to see the value of the global variable or use Xilinx Standard debugger (XMD) to see the global variables values.

No support for Linux Application debug from Xilinx Customized debugger.

Use Remote ARM Linux Application configuration to debug Linux Applications.

Deleting the last debugged application immediately (using Xilinx Customized TCF Debugger) using the Delete Project contents on disk option, causes an exception.

Workaround for this issue is to wait for 4 seconds after debugging, then try deleting the application.

Cannot debug MicroBlaze applications using Xilinx Customized TCF debugger.

Use Xilinx Standard debugger to debug the Microbalze applications.

TCF debugger by default starts at Application entry (main()).

To start debugging from program entry, check the Stop at program entry checkbox in the debug configuration.

Some of the variables listed in the Variables view do not show their values.

This is a known issue.

Frequently Asked Questions

Changing cable type in “Configure JTAG Settings” dialog box is prompting to terminate any active XMD-GDB debug sessions.

As XMD supports one cable type at a time, it is required to terminate any active launches.

Global variables are not displayed with Xilinx Customized TCF debugger.

Hover the mouse on the global variable to see the value of the global variable or use Xilinx Standard debugger (XMD) to see the global variables values.

No support for Linux Application debug from Xilinx Customized debugger.

Use Remote ARM Linux Application configuration to debug Linux Applications.

Cannot debug MicroBlaze applications using Xilinx Customized TCF debugger.

Use Xilinx Standard debugger to debug the MicroBlaze applications.

Cannot perform standalone profiling with Xilinx Customized TCF Debugger.

Use Xilinx Standard debugger to profile the Standalone applications.

Compilation happens properly but screen shows red marks indicating as if it failed due to errors.

CDT static code analysis is run on your source code by default. Some of the code analysis issues are reported as errors by default. You can switch Code Analysis Severity to Info from Error by going to Window > Preferences > C/C++ > Code Analysis.

Cannot perform standalone profiling with Xilinx Customized TCF Debugger.

Use Xilinx Standard debugger to profile the Standalone applications.

Why is it that I cannot set more than 5 breakpoints/watchpoints with the TCF Debugger?

Internally a breakpoint/watchpoint planted by user is treated as Hardware breakpoints in TCF debugger. For ARM there is a limitation of 6 Hardware breakpoints. Out of 6, one breakpoint is used for the single stepping, other breakpoint is used for stopping at main(). So users are left with four breakpoints. Software breakpoints (No limitation on number of breakpoints) support will be added in next release.

Cannot perform standalone profiling with Xilinx Customized TCF Debugger.

Use Xilinx Standard debugger to profile the Standalone applications.

Debug after re-launch does not stop on breakpoints.

SDK does processor reset on every debug session. The Interrupt Controller is not reset during Processor reset. Change the reset type to System Reset in the Device Initialization tab of debug configuration.

Not able to view the ps7 peripheral registers in TCF debugger.

By default these registers are not shown. Set an environment variable, HW_SERVER_ALLOW_PL_ACCESS, in the command prompt, then launch the hw_server.

FPGA need to program again after System Reset.

This is a change introduced with the latest silicon versions. If you are debugging PS+PL design and have the Reset Entire System option in the debug configuration, perform program FPGA when the debugger stops at application main().

Xilinx TCF debugger does not complain whether the FPGA is programmed or not.

This is a known issue.

Not able to connect to the board when Verbose mode is enabled in SDK XMD console.

Turn off verbose mode in SDK XMD console.

Build Project and Clean Project context menu options are missing for new BSP projects.

BSP projects uses Makebuilder to build BSP applications. When the Project > Build Automatically option is set, both Build Project and Clean Project are not visible, but any changes made to the BSP project triggers an auto-build. To see these context menu options for a BSP project, unset the Project > Build Automatically option.

I have installed cable drivers in my system. But launching hw_server gives a "missing FTD2XX.dll" error message.

If you never plugin the digilent cable, you can run into this issue. Plug-in the digilent cable; this copies the required libraries.
