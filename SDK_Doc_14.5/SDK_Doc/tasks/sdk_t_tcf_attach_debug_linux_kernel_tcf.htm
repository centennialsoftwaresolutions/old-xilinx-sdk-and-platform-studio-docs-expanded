Attach and Debug Linux Kernel Using Xilinx TCF Debugger

It is possible to debug the Linux Kernel using the Xilinx TCF debugger. Please follow the following steps to attach to the Linux Kernel running on target and debug the source code.

  1. Compile the kernel source using the following config options:
    1. CONFIG_DEBUG_KERNEL=y
    2. CONFIG_DEBUG_INFO=y
  2. Launch SDK.
  3. Open Debug perspective by clicking Window > Open Perspective > Debug.




  4. Open Debug Configurations by clicking Run > Debug Configurations.





  5. In the Debug Configurations dialog box, select the Xilinx TCF configuration type and click the New button. Name the configuration Zynq_Linux_Kernel_Debug.



  6. Unselect the Disconnect when last debug context exits check box and press the Debug button.




  7. Debugging will start and the processors are in running state as shown below.



  8. Click the button to suspend the processor. Debug starts in the Disassembly mode.



  9. Add Symbol file by right clicking ARM Cortex-A9 MPCore#0 and selecting Symbol Files.



  10. Click Add to add the Symbol file.



  11. Add vmlinux symbol file and click OK.




  12. You need to setup “Source Lookup”, if you built the code in Linux machine and try to run the debugger on Windows.



  13. Select the debug configuration “Zynq_Linux_Kernel_Debug” and right-click it to select Edit Source Lookup.








  14. Click Add and select Path Mapping.




  15. Add the Compilation path and local File system path by clicking Add.


  16. Successful source look up will take you to the source code debug.



  17. You can add function breakpoints by clicking the button in the Breakpoints view.




  18. Add a breakpoint at the start_kernel function.






  19. Press reset button, Zynq boots from the SD card and stops at the beginning of the kernel initialization.

    Note:     Linux kernel is always compiled with full optimizations and inlining enabled. Therefore:
    • Stepping through code might not work as expected due to the possible reordering of some instructions.
    • Some variables might be optimized out by the compiler and therefore not be available for the debugger.
