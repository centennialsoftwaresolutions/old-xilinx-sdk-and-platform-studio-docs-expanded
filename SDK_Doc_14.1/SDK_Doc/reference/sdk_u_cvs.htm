Using SDK with Revision Control Software

This section provides details on how to use SDK in conjunction with version control systems (VCS). The specific VCS used does not matter - the instructions are applicable across CVS, SVN, Mercurial, git, Perforce or anything else.

Files to be placed in version control

To reproduce an SDK application build, the following projects have to be saved in version control:

  • Hardware Platform Project
  • BSP project
  • Application project

Within the hardware project, the following files need to be saved:

  • system.xml - the hardware platform specification.
  • system.bit and system_bd.bmm - the bitstream and BMM files necessary for configuring the FPGA
  • .project - The Eclipse project file that contains project specific meta data.
Note: Typically, the Eclipse resource navigator does not show files that being with a period. This setting can be changed by changing the list of filter patterns to be applied. This is available in a drop down menu in the resource navigator view.

The following files present in the BSP project should be saved:

  • libgen.options - command line options provided to libgen
  • system.mss - The BSP specification.
  • Makefile - The BSP project's Makefile is not automatically generated. So this Makefile should be preserved.
  • .project, .sdkproject and .cproject - Eclipse and SDK meta-data files.

The following files present in the application project should be saved:

  • All source and header files.
  • Any other file user generated file like documentation files.
  • .project and .cproject - Eclipse meta-data files.

Organizing projects in directories

SDK (or Eclipse) projects can be created either inside a workspace, or in any other folder on the hard disk. Due to the various inter dependencies (application projects depend on BSPs and hardware projects, and BSPs depend on the hardware projects), SDK generated tool invocations have to point to files in their dependent projects. To make builds reproducible, it is highly advantageous if there are no absolute paths used in any SDK saved settings.

In order to get SDK to use relative paths, put all the dependent projects in a single folder. As long as the hardware, BSP and application projects are in the same folder, all the SDK generated command line invocations of various tools will use relative paths.

If software repositories are used to store drivers and libraries that are processed by libgen, then these repositories should also be placed in the same folder as the BSP project (which is the same folder as the application and h/w projects). If such a directory structure is maintained, libgen will use a relative path to access these software repositories.

The use of Eclipse variables (like ${WORKSPACE}), can lead to the use of absolute paths. So the use of such variables should be considered on a case by case basis.

Typical User Flow

Initial Setup

Eclipse provides many different team provider plugins. These plugins integrate various source control tools into Eclipse. Plugins exist for many different tools and can be obtained from eclipse.org. Users may choose to use these plugins for better integration, or they may perform source control operations outside the Eclipse GUI.

When starting from an existing project setup (projects already exist in source control), perform the following steps:

  1. Obtain projects from version control into local hard disk
  2. If the projects use a EDK repository for drivers or libraries, set that first in the workspace.
  3. Import the projects into the workspace. Make sure to deselect the "Copy project sources" option provided in the import wizard. This will leave the projects in the same place where they were initially checked out from source control.
  4. SDK will automatically build all the projects and produce the artifacts (libraries and executables).

When starting from scratch, simply follow the regular steps to create SDK applications, but make sure that all the projects are placed in the same folder (it can be inside or outside the workspace). Then check in the files specified above into source control.

Everyday Workflow

The main work is in the initial setup. Everyday workflow is typically just editing files, building and testing them, and finally checking them in. Simply follow the steps required by the version control tool to perform these steps.

Command line Builds

Eclipse projects can be built from the command line using GNU make. The steps differ based primarily on how the Makefiles will be generated.

Auto Generated Makefiles

By default, SDK auto generates Makefiles for the application project. However, these Makefiles are a generated artifact and should not be placed under version control. Users who let SDK generate Makefiles then have to invoke SDK/Eclipse in a headless mode to build their software. The build steps will typically look like the following:

  1. Invoke SDK in headless mode and import necessary projects.
  2. Invoke SDK in headless mode and build necessary projects. The Makefiles will be generated as part of this step, and the build artifacts will be generated.
See the SDK Command Line Flows section for more details on these steps.

User provided Makefiles

Users can also provide their own Makefiles for complete control of the build process. In such a case, they can instruct the GUI to not generate Makefiles, and invoke make with their Makefile to build the software. In such a case, building software from the command line is trivial as it just involves invoking make with the user Makefile.
